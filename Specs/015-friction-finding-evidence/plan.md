## 1. Goal & success criteria

Deliver the foundational **evidence substrate** the discover→interpret→gate loop is built on: two hand-editable, per-event files that carry machine-readable gate fields, validated on read, with a lifecycle that only ever moves deliberately.

**Roadmap success metrics:**
- A friction event **and** a finding **validate on read against their schema**; hand-edited frontmatter is accepted.
- The lifecycle `logged → candidate → codified → proven → closed` (with `watch`/`held` side states) is **representable and never transitions implicitly**.

**Deliverables:**
1. `EvidenceHeader` (shared fields) + `FrictionEvent` + `Finding` typed records; an `Evidence` protocol for uniform consumption.
2. `LifecycleState` enum + `EvidenceLifecycle` (legal-moves table + pure `canTransition`/`validate` validator).
3. The codec (`ConfigYAML` seam): split frontmatter/body, peek `schema:` → decode the typed record (`.reject` on duplicate keys), **validate**; encode a fresh record on create. (In-place *updates* are F30's, via swift-yaml `set`/`insert` — D6.)
4. `EvidenceError` domain-error enum (boundary-translatable later, like `CorpusError`).
5. Inline synthetic golden fixtures + unit tests (round-trip, tolerance, every failure mode, every lifecycle edge).

**Non-goals (explicit owners):** the `friction add|list|show|set-state|render` commands → **F30**; findings *mining* (producing findings from the corpus) → **triage / Phase 4**; the gates engine that *consumes* evidence → **Phase 5**; friction-log prose→structure migration → **F31**; loading a skill's whole evidence set off disk (directory walk) → **F30** (a thin I/O layer over this codec); checking that linked sessions/evals exist → later.

## 2. Scope

**In scope:** the pure model, the lifecycle + validator, the validation/dispatch, and the error type (`EDDCore`); the thin YAML text↔typed-record decode/create seam (`ConfigYAML`); and their tests.

**Out of scope:** any command-line surface; any filesystem access beyond "hand me the file text"; consuming/aggregating evidence (gates); generating the `friction-log.md` view (F30 `render`).

## 4. Lifecycle (design §7.3, D6)

States: `logged`, `candidate`, `codified`, `proven`, `closed` (main path) + `watch`, `held` (human-set side states). Modeled as one `LifecycleState` enum (illegal *values* unrepresentable in code; a hand-edited bogus value fails loud on read).

**Legal-moves table (Unknown 1 resolved 2026-07-17 → Option A + one reopen edge):**

| From | Allowed → |
|---|---|
| `logged` | `candidate`, `watch`, `held`, `closed` |
| `candidate` | `codified`, `watch`, `held`, `closed`, `logged` (revert) |
| `codified` | `proven`, `held`, `closed` |
| `proven` | `closed`, `held` (regressed) |
| `watch` | `logged`, `candidate`, `closed` |
| `held` | `logged`, `candidate`, `closed` |
| `closed` | `logged` (**reopen** — a recurring friction / re-detected finding re-enters the pipeline) |

Validator: `EvidenceLifecycle.canTransition(from:to:) -> Bool` (and a throwing `validate(from:to:)`), pure. **Self-transitions (`X → X`) are rejected** — encoding "never a silent no-op / never implicit". Future commands (F30 `set-state`, `next`, `eval new`, `iterate --mark`) call this one validator; none writes a status without passing it (the decision is pure; the file write is the command's, after — keeping side effects out of the decision).

**Reopen — the only exit from the terminal `closed` (Unknown 1):** `closed → logged` re-enters the pipeline when a resolved item recurs — a friction's third occurrence, or a finding the scorer re-detects — instead of spawning a duplicate that loses the corroboration count. (Not the *only* backward edge — `candidate → logged` and the `watch`/`held → logged`/`candidate` returns also step back — but the one **reopen** from the terminal state.) This is the standard reopen transition ([Jira reopen](https://support.atlassian.com/jira/kb/how-to-modify-your-workflow-to-allow-issues-to-be-reopened/); GitLab **reinstates** a reintroduced finding to triage), constrained to a single source (`closed`) as the state-machine guidance advises.

**Still excluded (reviewable):** **no direct `watch ↔ held`** (distinct human judgments — re-decide by returning to `logged`/`candidate` first, so the change is explicit and the reason re-supplied) and **no `codified`/`proven` → `logged`/`candidate` revert** (a regressed item goes to `held` (blocked) or `closed` (abandoned/then-reopen), never silently back-steps its forward progress — advancement stays auditable). If a real workflow needs either edge, add it to the table with a note; the validator is the single place it changes.

## 5. Codec & validation

**Path B (re-scoped 2026-07-17 — D6).** swift-yaml now ships **in-place editing** (`YAMLEditor.set`/`insert`) and **native duplicate-key rejection** (`DuplicateKeyStrategy.reject`), so F29 **reads into the typed records and only ever *creates* new files** — it never round-trips a loosely-typed map. That deletes the whole Path-A workaround stack: the neutral `[String: JSONValue]` map, the `JSONValue.init` reorder, the custom duplicate-key pre-scan, and the round-trip scalar-fidelity accounting are **gone**. The layer split (D4) stays: YAML↔model conversion lives in `ConfigYAML`; the models, dispatch, validation, and lifecycle stay pure in `EDDCore`.

**`ConfigYAML` seam** — mirrors `FrontmatterParser`/`ConfigLoader`; owns the serialization errors:
- `decode(_ text: String, filename: String? = nil) throws -> (evidence: any Evidence, body: String)`: split the leading `---\n … \n---\n` block; **peek `schema`** (a one-field decode) to choose the record; then `YAMLDecoder(duplicateKeyStrategy: .reject).decode(FrictionEvent.self | Finding.self, …)`. Type / enum / required-field checks come free from `Codable` decode; **duplicate keys are rejected by the library** (an event-driven scan → `duplicateKey(key:line:column:)`, so a doubled `state:` fails with a line number rather than silently last-winning). Unknown frontmatter keys are **ignored** on read (standard `Codable`, exactly as `SkillFrontmatter` does) — safe, because reads never write an existing file back (D3). Errors: `schema` absent/unknown → `unknownSchema`; a decode failure → mapped to `missingField`/`invalidValue` (the coding path names the field); bad delimiters / un-decodable YAML → `ConfigError.malformedFrontmatter`; a rejected duplicate → `ConfigError.duplicateKey`. `body` = everything after the closing `---`, minus the single file-terminating newline (POSIX §3.206 — it terminates the last line, not content), so `decode(encode(x)).body == x.body`.
- `encode(_ evidence: any Evidence, body: String) throws -> String` (**create only**): `YAMLEncoder` the record → frontmatter in **field-declaration order** (deterministic *and* human-readable — matches the design §7.3 example, so **no `.sortedKeys` and no example regeneration**), then wraps as `"---\n" + <yaml> + "\n---\n" + <body>` with a **trailing newline** (POSIX text-file convention — matches `RenderKit`). A fresh file has only known fields — nothing to preserve.

**Updating an existing file is F30's job, not F29's** — F30 uses swift-yaml **`YAMLEditor.set`** (change the status) + **`YAMLEditor.insert`** (add the reason / fill the eval link) **in place**, keeping comments, key order, quoting, and every human-added field byte-identical. That *is* how "never delete a human's edit" (D3) is delivered — by never re-serializing an existing file — which is why F29's codec needs no map or round-trip machinery.

**`EDDCore` validation (the checks `Codable` can't express):** `id` shape; the **`state_reason` conditional (D5)** — `state ∈ {held, watch, closed}` ⇒ non-empty, else `missingStateReason`; the **filename cross-check (F4)** — `Evidence.validate(filename: String? = nil)` (a protocol-extension method on the decoded record) takes the source filename (F30 passes `url.lastPathComponent`; standalone decode omits it), `idMismatch` when the stem ≠ `id`. `sessions` decodes as a strict `[String]` (a non-string element is a `Codable` decode error, never a silent drop). The **lifecycle** validator (`canTransition`) is a *write-time* check F30 calls when it changes a status (→ `illegalTransition`), not a read-time one.

**Error taxonomy — split by layer:** **`ConfigError`** (seam): `malformedFrontmatter`, `duplicateKey(key:line:column:)`. **`EvidenceError`** (EDDCore, pure): `unknownSchema`, `missingField`, `invalidValue`, `missingStateReason`, `idMismatch`, `illegalTransition`. The command boundary translates both to `EDDError` (seam → `.invalidArtifact`). *(The seam imports `EDDCore`, so it maps `Codable` decode failures into the pure `EvidenceError.missingField`/`invalidValue` types, raised at the edge.)*
